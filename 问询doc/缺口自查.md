## 3. P0-A：严格区分 position 读取失败与真实空仓

### 当前问题

当前 `dbt_get_positions()` 定义为：

```python
def dbt_get_positions(currency, kind="option"):
    return _call("GET", "/private/get_positions", {"currency": currency, "kind": kind}, "get_positions", _READ_RETRIES) or []
```

这会把以下两种完全不同的状态都变成 `[]`：

1. 交易所真实返回空持仓；
2. `_call()` 返回 `None`，例如网络/API/权限/Deribit error。

v3.1.5 的 `_settlement_reconcile_snapshot()` 会在到期 + 宽限后，根据“交易所无该 option position”把本地腿归零。因此，如果 position 读取失败被折叠为空列表，存在**误交割 finalize** 的风险。

### 修复要求

新增 strict 读取函数，保留旧函数兼容旧调用。

建议实现：

```python
def dbt_get_positions_strict(currency, kind="option"):
    """返回 list 表示读取成功；返回 None 表示读取失败。不得把失败折叠为空列表。"""
    return _call("GET", "/private/get_positions",
                 {"currency": currency, "kind": kind},
                 "get_positions", _READ_RETRIES)


def dbt_get_positions(currency, kind="option"):
    """兼容旧调用；新风控/交割/恢复路径不得使用此函数判断空仓。"""
    return dbt_get_positions_strict(currency, kind) or []
```

随后修改以下路径：

- `startup_recovery_check()`：使用 `dbt_get_positions_strict()`；若返回 `None`，设置 `RECOVERY_BLOCKED: OPTION_POSITION_QUERY_FAILED`，不得运行 settlement reconcile。
- `manage_cycle()`：使用 strict read；若返回 `None`，本轮不执行 `_settlement_reconcile_snapshot()`，并在 `ledger_detail` / `ctx` 标记 `OPTION_POSITION_QUERY_FAILED`。
- `_read_startup_positions()`：期权腿读取也必须 strict；对冲腿读取已经有 Binance `None` 阻断，但 Deribit future 也应 strict 化。

### 验收标准

- 当 option positions 读取失败时：
  - 不新增 `option_settlement_history`；
  - 不修改 `remaining_short_qty` / `long_remaining_qty`；
  - 不把 hedge target 强制置 0；
  - 面板显示 `OPTION_POSITION_QUERY_FAILED` 或等价 data gap；
  - 新开仓保持 fail-closed。
- 当 option positions 成功返回 `[]` 且已到期 + 超过 grace：才允许 settlement finalize。

### 必加测试

1. `_settlement_reconcile_snapshot()` 不应在 `option_positions=None` 时被调用；如果调用，也必须返回 `changed=False` / `reason=OPTION_POSITION_DATA_GAP`。
2. `manage_cycle()` 模拟 strict read 返回 `None`：快照数量不变。
3. `startup_recovery_check()` 模拟 strict read 返回 `None`：返回 `RECOVERY_BLOCKED`，不清仓、不归档。

## 5. P0/P1：CLOSED 归档后清恢复态，避免清完 hedge 后仍禁新开

### 当前问题

`_archive_closed()` 当前只做：

```python
_G(_CLOSED_HISTORY_KEY, hist[-50:])
_G(_POSITION_KEY, None)
ledger_set_state(S_CLOSED)
```

没有清 `_RECOVERY_KEY`。如果启动恢复曾设置：

```python
{"state": "ORPHAN_HEDGE_EMERGENCY", "allow_new_open": False}
```

那么对冲清理完成、两腿归零、仓位归档后，下一轮可能因为 `rec_ok=False` 落到 `RECOVERY_BLOCKED` 展示阶段，阻断新一轮计划。

### 修复要求

`_archive_closed()` 成功归档后，必须把恢复态置回 OK。

建议实现：

```python
def _clear_recovery_ok(reason, now_ms):
    verdict = {"state": "OK", "reasons": [], "allow_new_open": True,
               "cleared_reason": reason, "cleared_ts": now_ms}
    _G(_RECOVERY_KEY, verdict)
    return verdict
```

在 `_archive_closed()` 尾部：

```python
_clear_recovery_ok("POSITION_CLOSED_ARCHIVED", now_ms)
```

同时建议清理 hedge policy pending 状态，避免旧 position id 的 pending 干扰显示：

```python
_G(_HEDGE_POLICY_STATE_KEY, _hedge_policy_default_state(None))
```

注意：如果有未完成 hedge pending order，不得归档 CLOSED。当前 CLOSED 条件用 `perp_qty==0`，但还应确认 V313 pending 已清：

```python
not (_G(_HEDGE_POLICY_STATE_KEY) or {}).get("pending_order_id")
```

### 验收标准

- orphan hedge 清理完成后，下一轮 `run_cycle()` 不应因旧 `_RECOVERY_KEY.allow_new_open=False` 卡住。
- `_POSITION_KEY=None`、`ledger_state=S_CLOSED`、`_RECOVERY_KEY.state=OK` 时，若 manual context valid，应进入 `PLAN_MENU_READY` / `HARD_APPROVAL_WAIT`，而不是 `RECOVERY_BLOCKED`。
- 如果还有 hedge pending order，不能归档 CLOSED。

### 必加测试

1. 初始 `_RECOVERY_KEY=ORPHAN_HEDGE_EMERGENCY`，snap 两腿 0，perp 0：调用 `_archive_closed()` 后 recovery OK。
2. `_RECOVERY_KEY=RECOVERY_BLOCKED` 且人为调用 `_archive_closed()`：除非 closed 条件满足，否则不得清 recovery。
3. hedge pending 存在：不归档 CLOSED，不清 recovery。

## 7. P1：无快照 orphan hedge 的处理策略必须明确

### 当前问题

当 `_POSITION_KEY` 丢失、交易所无 option 风险、但 Binance/Deribit 仍有 perp hedge 时，`evaluate_startup_recovery()` 会返回：

```python
ORPHAN_HEDGE_EMERGENCY / allow_new_open=False
```

但 `run_cycle()` 在 `has_pos=False` 时不会调用 `manage_cycle()`，最终进入恢复阻塞展示。这样不会自动清理孤儿对冲。

### 必须二选一实现

#### 方案 A：安全阻塞，不自动清外部仓位

适合担心误关手工 perp 的场景。要求：

- `run_cycle()` phase 明确显示 `ORPHAN_HEDGE_MANUAL_CLEANUP_REQUIRED`，不要伪装成普通 `RECOVERY_BLOCKED`。
- 面板显示：当前 perp qty、venue、instrument、建议人工 reduce-only 方向。
- 新开仓保持禁止，直到人工清理后启动恢复重新变 OK。

#### 方案 B：带所有权证据的 orphan-only reduce-only cleanup

适合希望机器人自动清理本策略遗留 hedge 的场景。要求：

- 只有存在策略所有权证据时自动执行，例如：
  - `_HEDGE_POLICY_STATE_KEY.position_id` 与 closed/历史 position 匹配；或
  - `_POSITION_KEY` 刚刚被 settlement finalize 过；或
  - hedge order history 中有本策略 label / ROBOT_ID；
  - 或显式配置 `ALLOW_ORPHAN_HEDGE_ONLY_CLEANUP=True`。
- 只允许 reduce-only，不允许 open/increase。
- side 由真实 perp qty 符号决定：
  - qty > 0：卖出 reduce-only；
  - qty < 0：买入 reduce-only。
- 清到 abs(qty) <= min_trade/epsilon 后，清 `_RECOVERY_KEY` 为 OK。

建议函数：

```python
def manage_orphan_hedge_only(now_ms):
    # no _POSITION_KEY; no option positions; only reduce existing perp to zero
```

### 验收标准

无论选 A 或 B，都不能继续表现为“无说明地 RECOVERY_BLOCKED”。必须让交易员/日志知道下一步是什么。

### 必加测试

1. 无 snap + 无 option + perp != 0：进入明确 phase。
2. 方案 A：不下单，显示人工清理方向。
3. 方案 B：只发 reduce-only，方向与 perp qty 相反；清完后 recovery OK。
4. 无 snap + option position 存在：仍然 RECOVERY_BLOCKED，不得自动处理。

## 8. P1：Ledger / Display 补 settlement 与 final PnL 展示

### 当前问题

`_build_ledger_detail()` 目前只展示：入场、退出、保护回收、对冲历史条数、对账和恢复状态。没有展示 settlement event count、settlement pnl status、final option PnL status。


如果 `settlement_pnl_status in (DATA_GAP, ESTIMATED)`，显示必须明显区别于 OK。

## 9. P2：交割后风险评估降噪

### 当前问题

短腿已经 settlement 后，`_evaluate_position_risk_now()` 仍可能尝试读过期短腿 quote，进而返回 `RISK_MARKET_DATA_GAP`。这不一定阻断收口，但会污染面板和日志。

### 修复要求

```python
if not snap or (snap.get("remaining_short_qty") or 0.0) <= 1e-12 \
        or snap.get("settlement_state") in ("SHORT_SETTLED", "BOTH_LEGS_SETTLED", "SETTLED"):
    return {
        "tail_risk_state": None,
        "market_data_gap": False,
        "current_risk": {},
        "reason_codes": ["OPTION_SETTLED_NO_SHORT_RISK"],
    }
```

### 验收标准

- 短腿已归零后不再显示 `RISK_MARKET_DATA_GAP`。
- hedge orphan cleanup 仍然按 `_evaluate_hedge()` 执行，不依赖 risk package。

## 10. 测试矩阵：必须覆盖到这些场景

### 10.1 交割状态机

| 场景 | 输入 | 预期 |
|---|---|---|
| 未到期 | now < expiry + grace，positions 空 | 不归零，不写 settlement event |
| 到期但 position 读取失败 | strict read None | 不归零，RECOVERY_BLOCKED/DATA_GAP |
| 到期且 short absent | short qty >0，actual 无 short | short qty -> 0，写 short settlement event |
| 到期且 long absent | long qty >0，actual 无 long | long qty -> 0，写 long settlement event |
| 两腿均 absent | short/long qty >0，actual 均无 | state=BOTH_LEGS_SETTLED |
| 交易所有 unexpected option | snap 无该腿但 actual 有 | RECOVERY_BLOCKED 或 reconcile mismatch，不自动归档 |

### 10.2 交割 PnL

| 场景 | 预期 |
|---|---|
| OTM short | settlement cashflow 0 |
| ITM short | settlement cashflow 负数 |
| ITM long | settlement cashflow 正数 |
| vertical both ITM | net settlement cashflow 与腿宽方向一致 |
| settlement price 缺失 | status=DATA_GAP，不伪造 0 |
| fallback index | status=ESTIMATED，不标 COMPUTED |

### 10.3 对冲收口

| 场景 | 预期 |
|---|---|
| short settled + perp short | target=0，side=buy，reduce_only=True |
| short settled + perp long | target=0，side=sell，reduce_only=True |
| short settled + no perp | 不发 hedge 单，允许后续归档 |
| hedge position read fail | 不误发单，显示 HEDGE_POSITION_DATA_GAP |
| V313 pending active | 不重复下第二单 |
| V313 stale partial fill | 记录 hedge_execution_history，清 pending 或继续单飞 |

### 10.4 归档/恢复

| 场景 | 预期 |
|---|---|
| 两腿 0 + perp 0 + no pending | `_archive_closed()`，清 `_POSITION_KEY`，state=CLOSED，recovery=OK |
| 两腿 0 + perp !=0 | 不 CLOSED，先 ORPHAN_HEDGE_EMERGENCY |
| 两腿 0 + perp 0 + pending hedge | 不 CLOSED |
| CLOSED 后下一轮 | 进入 PLAN_MENU_READY/HARD_APPROVAL_WAIT，不进入 RECOVERY_BLOCKED |
| no snap + perp !=0 | 明确 orphan-only 策略，不静默阻塞 |

### 10.5 预算

| 场景 | 预期 |
|---|---|
| current positions 读取失败 | projected budget BLOCK |
| account summary 读取失败 | projected budget BLOCK |
| 已有 1 个 open position，limit=1 | BLOCK |
| proposed 缺 margin/greeks | BLOCK，不默认 0 |
| 空账户 + 数据完整 | ALLOW |

### 10.6 Display / Ledger

| 场景 | 预期 |
|---|---|
| settlement event exists | ledger table 有交割记账行 |
| final pnl DATA_GAP | data_quality_state 非 OK |
| settled short | 风险表不显示 RISK_MARKET_DATA_GAP |
| protection recovery sold | ledger 展示 realized_protection_recovery_value |

## 13. 最终验收定义

完成后，应能明确回答以下问题且均为“是”：

1. 交易所 position 读取失败时，是否不会误判期权已交割？
2. 期权腿交割后，是否会记录交割现金流，而不只是记录“腿消失”？
3. 保护腿主动卖出后，是否会计入净回收价值？
4. `closed_history` 是否包含 option-only final PnL 及其状态？
5. short 已交割且 perp 存在时，是否只触发 reduce-only hedge 清理？
6. hedge 清理完成并 CLOSED 后，恢复态是否回到 OK，不再阻断新计划？
7. 账户已有持仓或账户数据读取失败时，预算门是否 fail-closed？
8. 交割后面板是否能看出 settlement/final pnl 是否完整？
9. 所有上述场景是否有本地单测覆盖？

## 14. 不要做的事

- 不要把 `settlement_pnl_ccy=None` 当成已完成记账。
- 不要把 position read failure 当成 `[]`。
- 不要用当前 index price 伪装成权威 settlement price；fallback 必须标 `ESTIMATED`。
- 不要在没有所有权证据时静默关闭无快照的外部 perp 仓位。
- 不要在组合预算中把缺失 gamma/vega/margin 当 0。
- 不要因为交割收口需要而允许任何 hedge open/increase；交割后只允许 reduce/unwind。
- 不要在 hedge pending 仍存在时归档 CLOSED。
